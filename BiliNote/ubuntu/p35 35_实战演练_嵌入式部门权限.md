## 实战目标

模拟一个嵌入式开发部门的权限管理场景，具体要求如下：

- 在 `/opt` 目录下创建名为 `嵌入式` 的工作目录。
- 部门领导“张总”拥有该目录及内部文件的 **全部权限（rwx）**。
- 部门开发人员（如“小张”）属于 `嵌入式` 用户组，对目录和文件具有 **读 + 执行权限（rx）**，但 **无写权限**。
- 其他用户（非本部门）仅对目录有 **读 + 执行权限（rx）**（可进入并列出内容），对文件仅有 **读权限（r）**，**无执行权限**。

> 注：视频中“签入式”“潜入式”等均为“嵌入式”的口误或字幕错误，统一修正为“嵌入式”。

## 步骤一：创建用户与组

### 1. 创建部门领导用户：`张总`
```bash
sudo adduser 张总
# 密码设为 123456（仅用于演示）
```

### 2. 赋予 `张总` sudo 权限
```bash
sudo usermod -aG sudo 张总
# 注意：先写组名（sudo），再写用户名（张总）
```

### 3. 创建开发人员用户：`小张`
```bash
sudo adduser 小张
# 密码同样设为 123456
# 不赋予 sudo 权限
```

### 4. 创建部门用户组：`嵌入式`
```bash
sudo addgroup 嵌入式
```

### 5. 将 `小张` 加入 `嵌入式` 组（作为附加组）
```bash
sudo usermod -aG 嵌入式 小张
```

> ✅ 此时：
> - `张总`：普通用户 + sudo 权限
> - `小张`：普通用户 + 属于 `嵌入式` 组
> - `嵌入式`：新创建的用户组

## 步骤二：创建嵌入式工作目录

### 1. 在 `/opt` 下创建目录
```bash
cd /opt
sudo mkdir 嵌入式
```

### 2. 修改目录所有者与所属组
```bash
sudo chown 张总:嵌入式 嵌入式
```

> ⚠️ 注意：普通用户（即使是文件创建者）**无法自行修改文件/目录的所有者或所属组**，必须使用 `sudo`。

## 步骤三：设置目录权限（755）

### 权限设计依据
- **所有者（张总）**：rwx → `7`
- **所属组（嵌入式组成员，如小张）**：rx → `5`
- **其他用户**：rx → `5`（允许进入目录并列出内容）

> 📌 关键点：**目录的“执行（x）”权限 = “进入目录”的权限**。若无 `x`，即使有 `r` 也无法 `cd` 进入或 `ls` 列出内容。

### 设置命令
```bash
sudo chmod 755 /opt/嵌入式
```

### 验证：其他用户（如 `itguigu`）尝试访问
```bash
ls /opt/嵌入式    # 可以看到文件列表（因有 r + x）
cd /opt/嵌入式    # 可以进入（因有 x）
```

> ❌ 若权限设为 `754`（其他用户无 `x`）：
> - `cd /opt/嵌入式` → **Permission denied**
> - `ls /opt/嵌入式` → **Permission denied**

## 步骤四：设置文件权限（754）

### 创建测试文件
以 `张总` 身份操作：
```bash
cd /opt/嵌入式
echo "嵌入式机密" > 机密文件.txt
```

### 权限设计
- **所有者（张总）**：rwx → `7`
- **所属组（嵌入式组）**：rx → `5`（可读、可执行脚本）
- **其他用户**：r → `4`（仅可读，不可执行）

### 设置权限与所属组
```bash
# 先确保文件属于正确组
sudo chown 张总:嵌入式 机密文件.txt

# 再设置权限
chmod 754 机密文件.txt
```

> ⚠️ 必须先用 `sudo chown` 修改所属组，否则 `小张` 无法获得组权限。

## 关键权限机制说明

### 1. 目录权限 vs 文件权限
| 类型   | 读（r）         | 写（w）           | 执行（x）         |
|--------|------------------|--------------------|--------------------|
| 目录   | 列出内容 (`ls`) | 创建/删除文件      | 进入目录 (`cd`)    |
| 文件   | 查看内容         | 修改内容           | 作为程序执行       |

### 2. 权限数字含义
- `7` = rwx = 4+2+1
- `5` = r-x = 4+0+1
- `4` = r-- = 4+0+0

### 3. 修改所有者/组的限制
- **只有 root 或具有 CAP_CHOWN 能力的用户**才能修改文件的所有者。
- 普通用户即使拥有文件，也不能将所有权转移给他人。

## AI 总结

本节通过一个典型的嵌入式开发部门权限管理场景，完整演示了 Linux 用户、组、目录与文件权限的综合应用。核心要点包括：

1. **用户与组的创建与关联**：使用 `adduser`、`addgroup` 和 `usermod -aG` 构建组织结构。
2. **所有权与权限分离**：通过 `chown` 设置所有者/组，`chmod` 设置访问权限。
3. **目录权限的特殊性**：`x` 权限决定能否进入目录，是权限设计的关键。
4. **权限最小化原则**：开发人员有读/执行权但无写权，保障代码安全；外部人员仅可读。
5. **root 权限的必要性**：修改文件所有者必须使用 `sudo`，体现系统安全机制。

该实战不仅巩固了权限命令的使用，更展示了如何在真实团队协作中设计合理的访问控制策略。
