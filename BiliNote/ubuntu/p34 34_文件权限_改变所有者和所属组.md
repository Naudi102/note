## 背景与问题场景

在 Linux 系统（如 Ubuntu）中，某些系统目录（例如 `/opt`）默认仅允许 **root 用户**进行写操作。普通用户若需在此类目录下安装软件或创建文件夹，会因权限不足而失败。

- `/opt` 是一个常用于存放**第三方软件**或**用户自定义安装内容**的目录。
- 默认情况下，该目录的所有者为 `root`，其他用户无写权限。

## 使用 root 创建文件的问题

当使用 `sudo` 命令（如 `sudo mkdir software`）在 `/opt` 下创建目录时：

- 新建目录的**所有者（owner）** 和 **所属组（group）** 均为 `root`。
- 即使目录已成功创建，普通用户（如 `atguigu`）仍无法在其中创建或修改文件，因为权限未开放。

示例：
```bash
sudo mkdir /opt/software
ls -ld /opt/software
# 输出：drwxr-xr-x 2 root root ... /opt/software
```

此时普通用户尝试在 `/opt/software` 中创建文件会失败。

*![](http://localhost:8000/static/screenshots/screenshot_1f0e25e70-5b1e-48ee-ad92-5eb2c64568b1.jpg)

## 不推荐的解决方案：777 权限

一种“简单粗暴”的解决方式是赋予目录 `777` 权限：

```bash
sudo chmod 777 /opt/software
```

- **效果**：所有用户（所有者、组、其他）都拥有读、写、执行权限。
- **风险**：完全丧失权限控制，相当于“裸奔”，存在严重安全风险。
- **结论**：**不推荐**在生产环境或常规使用中设置 `777` 权限。

## 推荐做法：修改所有者（chown）

更安全、合理的做法是将目录的**所有者**变更为目标用户（如 `atguigu`）：

```bash
sudo chown atguigu /opt/software
```

- 执行后，`atguigu` 成为该目录的所有者。
- 若目录权限为默认的 `755`（所有者可读写执行，组和其他只读执行），则 `atguigu` 可正常在其中创建、删除文件。
- 这是**标准且安全**的权限管理实践。

> 经典目录权限：`755`（rwxr-xr-x）

若目录下已有子目录/文件，需递归修改所有内容的所有者：

```bash
sudo chown -R atguigu /opt/software
```

## 修改所属组（chgrp 与 chown）

除了修改所有者，有时也需要修改**所属组**：

### 方法一：使用 `chgrp`
```bash
sudo chgrp xitian /opt/software
```

### 方法二：使用 `chown`（更常用）
`chown` 命令也可用于修改组，语法更统一，无需额外记忆 `chgrp`。

## 同时修改所有者和所属组

`chown` 支持**一次性修改所有者和所属组**，格式为：

```bash
sudo chown 用户名:组名 文件或目录
```

示例：
```bash
sudo chown atguigu:xitian /opt/moto
```

- 将 `/opt/moto` 的所有者设为 `atguigu`，所属组设为 `xitian`。
- 操作后，`atguigu` 拥有完整权限（rwx），`xitian` 组成员拥有 `rx` 权限（可进入目录、列出内容）。

## 权限模型说明

- **所有者（Owner）**：对文件/目录拥有最高控制权，通常由创建者或通过 `chown` 指定。
- **所属组（Group）**：用于将多个用户归为一组，统一授予权限。
- **其他用户（Others）**：系统中既非所有者也非组成员的用户。

典型安全配置：
- 目录权限设为 `755`
- 所有者为实际使用者（如开发用户）
- 所属组可设为项目组（如 `xitian`）
- 避免开放 `777` 或过度放宽“其他用户”权限
