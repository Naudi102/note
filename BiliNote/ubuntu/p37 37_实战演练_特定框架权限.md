## 背景与目标

在 Linux 系统（如 Ubuntu）中，许多软件（尤其是数据库类服务）在安装时会自动创建专属用户，并将相关数据目录的所有权分配给该用户。这类目录通常**不需要其他用户进行编辑或执行操作**，只需通过服务接口调用即可。

本节实战演练旨在演示如何为不同类型的软件框架（如 MySQL、Redis）设置合适的目录和文件权限，以实现：
- 安全隔离
- 功能可用性
- 防止误操作（如意外删除或新增文件）

## 场景一：仅需普通用户使用的服务（如 MySQL）

### 核心需求
- 软件（如 `mysql`）由专属用户运行。
- 其他用户**仅需读取和进入目录**，无需写入或执行内部文件。
- 服务启动后，普通用户通过客户端访问，不直接操作数据文件。

### 实施步骤
1. **创建专属用户（若未自动创建）**：
   ```bash
   sudo useradd mysql
   ```
2. **在 `/opt` 下创建软件目录**：
   ```bash
   sudo mkdir /opt/mysql
   ```
3. **设置所有权**：
   ```bash
   sudo chown mysql:mysql /opt/mysql
   ```
4. **设置目录权限为 `755`**：
   - 所有者（mysql）：`rwx`（读、写、执行）
   - 组用户和其他用户：`r-x`（读 + 执行）
   - **注意**：对目录而言，“执行”权限（`x`）意味着“可进入”。

### 权限说明
- `755` = `rwxr-xr-x`
- 其他用户可进入目录并读取内容，但**不能创建、删除或修改文件**。
- 这是**服务类软件的标准权限配置**。

## 场景二：允许修改内部文件但禁止增删文件（如 Redis）

### 核心需求
- 某些软件（如 `redis`）需要普通用户能**修改配置文件内容**，但**不能新增或删除文件**。
- 例如：运维人员需调整 `redis.conf`，但不能误删日志或数据文件。

### 实施步骤
1. **创建用户和目录**：
   ```bash
   sudo useradd redis
   sudo mkdir /opt/redis
   sudo chown redis:redis /opt/redis
   ```
2. **设置目录权限为 `755`**：
   ```bash
   sudo chmod 755 /opt/redis
   ```
   - 其他用户可进入（`x`）和列出内容（`r`），但**无写权限（`w`）** → 无法新增/删除文件。
3. **对需修改的文件单独设置权限（如 `redis.txt`）**：
   ```bash
   sudo chmod 666 /opt/redis/redis.txt  # 或 777，若需执行
   ```
   - `666`：所有用户可读写（`rw-rw-rw-`）
   - `777`：所有用户可读、写、执行（`rwxrwxrwx`）——适用于脚本或可执行配置

### 关键行为验证
- **其他用户**：
  - ✅ 可 `cd /opt/redis`
  - ✅ 可 `cat redis.txt` 或 `vim redis.txt`（若文件权限为 `666`/`777`）
  - ❌ 无法 `touch new.txt`（目录无 `w` 权限）
  - ❌ 无法 `rm redis.txt`（目录无 `w` 权限，即使文件可写）

> 💡 **重要提示**：目录的“写”权限控制**是否能增删文件**，而文件自身的权限控制**是否能读写内容**。两者独立。

## 关键权限规则总结

| 场景 | 目录权限 | 文件权限 | 用户行为 |
|------|--------|--------|--------|
| 服务类软件（MySQL） | `755` | 默认（由服务管理） | 仅读取，不可修改内容或结构 |
| 可配置软件（Redis） | `755` | `666` 或 `777`（按需） | 可修改指定文件内容，但不可增删文件 |

### 权限数字含义回顾
- `r` = 4, `w` = 2, `x` = 1
- 目录的 `x` = 可进入
- 目录的 `w` = 可增删文件（与文件自身权限无关）

### 最佳实践建议
- **避免滥用 `777`**：仅在必要时对特定文件开放，而非整个目录。
- **优先使用专属用户**：软件应以最小权限用户运行，提升系统安全性。
- **目录权限通常为 `755` 或 `750`**，文件权限按需精细控制。
